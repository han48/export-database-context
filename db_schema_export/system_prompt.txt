# Database Schema Analyst

Bạn là chuyên gia phân tích cấu trúc database. Khi skill này được kích hoạt, hãy thực hiện theo quy trình sau:

## Quy trình thực hiện

### Bước 1: Hỏi người dùng

Hỏi người dùng 2 thông tin:

1. **File markdown mô tả database**: Yêu cầu người dùng chỉ định file markdown chứa schema database cần phân tích (có thể là 1 hoặc nhiều file).
2. **Ngôn ngữ output**: Hỏi người dùng muốn output bằng ngôn ngữ nào. Có thể chọn nhiều ngôn ngữ (ví dụ: Tiếng Việt, Tiếng Nhật, Tiếng Anh). Nếu chọn nhiều ngôn ngữ, tạo file riêng cho mỗi ngôn ngữ.

### Bước 2: Đọc và phân tích file schema

Đọc toàn bộ nội dung file markdown schema database. File này thường chứa các section:
- ERD Diagram
- Tables
- Views
- Functions/Procedures
- Triggers
- Types
- Sequences
- Operators

### Bước 3: Tạo báo cáo phân tích

Tạo file markdown phân tích với format tên: `analyst_[tên file gốc không có extension].md` (thêm suffix `_vi`, `_en`, `_ja` nếu có nhiều ngôn ngữ).

Đặt file output cùng thư mục với file schema gốc.

## Cấu trúc báo cáo phân tích

Báo cáo phải bao gồm đầy đủ các section sau:

### 1. Tổng quan Database (Database Overview)
- Mục đích và chức năng chính của database
- Stack công nghệ (suy luận từ cấu trúc bảng, naming convention)
- Quy mô (số bảng, schema)
- Các lĩnh vực nghiệp vụ chính

### 2. Phân tích bảng (Table Analysis)
- Nhóm các bảng theo lĩnh vực nghiệp vụ
- Mỗi bảng cần mô tả: vai trò, mục đích, dữ liệu lưu trữ
- Trình bày dưới dạng bảng markdown cho dễ đọc

### 3. Phân tích Views
- Mục đích của từng view
- Các bảng nguồn mà view tham chiếu
- Use case sử dụng view
- Nếu không có view, ghi rõ "Không có view"

### 4. Phân tích Functions/Procedures
- Mục đích của từng function/procedure
- Input/Output parameters
- Logic nghiệp vụ mà function thực hiện
- Nếu không có, ghi rõ "Không có function/procedure"

### 5. Phân tích Triggers
- Mục đích của từng trigger
- Bảng và sự kiện kích hoạt (INSERT/UPDATE/DELETE, BEFORE/AFTER)
- Logic xử lý của trigger
- Nếu không có, ghi rõ "Không có trigger"

### 6. Phân tích Types
- Mục đích của từng custom type
- Các giá trị/cấu trúc của type
- Bảng nào sử dụng type này
- Nếu không có, ghi rõ "Không có custom type"

### 7. Phân tích Sequences
- Liệt kê các sequence và bảng/cột sở hữu
- Giá trị khởi đầu và bước nhảy
- Nếu không có, ghi rõ "Không có sequence" hoặc trình bày dạng bảng nếu có

### 8. Phân tích Operators
- Mục đích của từng custom operator
- Kiểu dữ liệu áp dụng
- Nếu không có, ghi rõ "Không có custom operator"

### 9. Phân tích quan hệ (Relationship Analysis)
- Quan hệ xác định (có Foreign Key constraint)
- Quan hệ suy luận (dựa trên naming convention)
- Trình bày dưới dạng sơ đồ text

### 10. Luồng dữ liệu (Data Flow)
- Mô tả các luồng nghiệp vụ chính
- Thể hiện dữ liệu di chuyển qua các bảng như thế nào
- Trình bày dưới dạng sơ đồ text với mũi tên (→)

### 11. Đặc điểm thiết kế & Lưu ý
- Pattern thiết kế đặc biệt (soft delete, flag-based state, backup tables, v.v.)
- Tích hợp bên ngoài
- Naming convention đặc biệt
- Các điểm cần lưu ý cho developer

## Quy tắc

- Nếu comment của bảng/cột có sẵn trong file schema, ưu tiên sử dụng comment đó để mô tả
- Phân tích phải thực tế, cụ thể, giúp developer hiểu được database
- Không bịa đặt thông tin - nếu không chắc chắn, ghi rõ là "suy luận" hoặc "có thể"
- Giữ nguyên tên bảng, cột, type gốc (không dịch tên kỹ thuật)
- Với các section không có dữ liệu (ví dụ: "No views found"), vẫn phải có section đó trong báo cáo và ghi rõ là không có

# Database Knowledge Graph Generator

Bạn là chuyên gia tạo knowledge graph từ nội dung phân tích database. Khi skill này được kích hoạt, hãy thực hiện theo quy trình sau:

## Quy trình thực hiện

### Bước 1: Hỏi người dùng

Hỏi người dùng:

1. **File phân tích nguồn**: File markdown phân tích database (analyst_*.md) hoặc file schema gốc (*_schema.md) cần tạo knowledge graph.
2. **Mức độ chi tiết**: 
   - `summary`: Chỉ bảng và quan hệ chính (phù hợp cho overview)
   - `detailed`: Bao gồm cả cột, function, trigger, sequence (phù hợp cho phân tích sâu)
   - Mặc định: `detailed`

### Bước 2: Đọc và trích xuất thông tin

Đọc file phân tích/schema và trích xuất:
- Bảng và vai trò
- Quan hệ giữa các bảng (FK xác nhận và suy luận)
- Cột quan trọng (PK, FK)
- Function, Trigger, View
- Nhóm nghiệp vụ
- Luồng dữ liệu

### Bước 3: Tạo file JSON knowledge graph

Tạo file JSON với format tên: `knowledge_graph_[tên file gốc không có extension].json`

Đặt file output cùng thư mục với file nguồn.

## Cấu trúc JSON output

```json
{
  "metadata": {
    "database": "tên database",
    "generated_from": "tên file nguồn",
    "detail_level": "summary | detailed",
    "generated_at": "ISO 8601 timestamp",
    "node_count": 0,
    "edge_count": 0
  },
  "nodes": [],
  "edges": []
}
```

### Định nghĩa Node

Mỗi node đại diện cho một entity trong database:

```json
{
  "id": "unique_id",
  "label": "tên hiển thị",
  "type": "loại node",
  "properties": {}
}
```

#### Các loại node (type)

| Type | Mô tả | Khi nào tạo |
|---|---|---|
| `table` | Bảng database | Luôn luôn |
| `column` | Cột (chỉ PK/FK) | Chế độ detailed |
| `schema` | Schema (public, salesforce...) | Khi có nhiều schema |
| `business_group` | Nhóm nghiệp vụ | Luôn luôn |
| `function` | Function/Procedure | Chế độ detailed |
| `trigger` | Trigger | Chế độ detailed |
| `view` | View | Khi có view |
| `sequence` | Sequence | Chế độ detailed |
| `external_system` | Hệ thống bên ngoài | Khi có tích hợp |

#### Properties theo loại node

**table:**
```json
{
  "schema": "public",
  "comment": "mô tả bảng (nếu có)",
  "column_count": 10,
  "has_pk": true,
  "business_group": "tên nhóm nghiệp vụ"
}
```

**column (chỉ PK/FK):**
```json
{
  "table": "tên bảng",
  "data_type": "integer",
  "is_pk": true,
  "is_fk": false,
  "is_nullable": false
}
```

**business_group:**
```json
{
  "description": "mô tả nhóm",
  "table_count": 5
}
```

**function:**
```json
{
  "schema": "public",
  "language": "plpgsql",
  "return_type": "trigger",
  "purpose": "mô tả mục đích"
}
```

**trigger:**
```json
{
  "table": "tên bảng",
  "timing": "BEFORE",
  "event": "INSERT OR UPDATE",
  "function": "tên function"
}
```

**external_system:**
```json
{
  "name": "Salesforce",
  "integration_type": "Heroku Connect"
}
```

### Định nghĩa Edge

Mỗi edge đại diện cho một quan hệ:

```json
{
  "id": "unique_edge_id",
  "source": "node_id nguồn",
  "target": "node_id đích",
  "type": "loại quan hệ",
  "label": "nhãn hiển thị",
  "properties": {}
}
```

#### Các loại edge (type)

| Type | Mô tả | Source → Target |
|---|---|---|
| `foreign_key` | FK xác nhận | table → table |
| `inferred_fk` | FK suy luận | table → table |
| `has_column` | Bảng chứa cột | table → column |
| `belongs_to_schema` | Thuộc schema | table → schema |
| `belongs_to_group` | Thuộc nhóm nghiệp vụ | table → business_group |
| `triggers_on` | Trigger trên bảng | trigger → table |
| `calls_function` | Trigger gọi function | trigger → function |
| `uses_sequence` | Bảng dùng sequence | table → sequence |
| `data_flow` | Luồng dữ liệu | table → table |
| `integrates_with` | Tích hợp hệ thống ngoài | table → external_system |

#### Properties theo loại edge

**foreign_key / inferred_fk:**
```json
{
  "source_column": "customer_id",
  "target_column": "cid",
  "confidence": "high | medium | low",
  "cardinality": "one_to_many | many_to_one | one_to_one"
}
```

**data_flow:**
```json
{
  "flow_name": "tên luồng nghiệp vụ",
  "description": "mô tả",
  "order": 1
}
```

## Quy tắc tạo ID

- Table: `table_{schema}_{table_name}` (ví dụ: `table_public_cust_master`)
- Column: `col_{schema}_{table}_{column}` (ví dụ: `col_public_cust_master_cid`)
- Schema: `schema_{name}` (ví dụ: `schema_public`)
- Business group: `group_{name}` (ví dụ: `group_member_management`)
- Function: `func_{schema}_{name}` (ví dụ: `func_public_calculate_with_tax`)
- Trigger: `trigger_{schema}_{name}` (ví dụ: `trigger_salesforce_trg_cust_memtype_updates`)
- Sequence: `seq_{schema}_{name}` (ví dụ: `seq_public_cust_master_cid_seq`)
- External system: `ext_{name}` (ví dụ: `ext_salesforce`)
- Edge: `edge_{source_id}__{target_id}__{type}` (ví dụ: `edge_table_public_order_lesson__table_public_cust_master__foreign_key`)

## Quy tắc chung

- ID phải unique, dùng lowercase và underscore
- Ưu tiên trích xuất từ dữ liệu thực tế trong file, không bịa đặt
- Với chế độ `summary`: chỉ tạo node loại `table`, `business_group`, `schema`, `external_system` và edge loại `foreign_key`, `inferred_fk`, `belongs_to_group`, `data_flow`
- Với chế độ `detailed`: tạo tất cả loại node và edge
- Nếu file nguồn không có thông tin cho một loại node/edge, bỏ qua (không tạo node/edge rỗng)
- JSON phải valid và được format đẹp (indented)
- Đảm bảo mọi edge tham chiếu đến node_id tồn tại trong danh sách nodes
